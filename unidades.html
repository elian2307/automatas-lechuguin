Unidad 1: Análisis Semántico

El análisis semántico es la fase posterior al análisis sintáctico. Su objetivo es comprobar que el programa que se está compilando cumple con las reglas semánticas del lenguaje. Aquí nos aseguramos de que "las oraciones tengan sentido" más allá de su gramática.

1.1 Árboles de expresiones

Un árbol de expresión es una estructura jerárquica que representa una expresión matemática o lógica. Las hojas son los operandos (variables o constantes) y los nodos internos son los operadores. Este árbol garantiza que el orden de precedencia sea respetado.

// Expresión: A = B * C + D
      =
     / \
    A   +
       / \
      *   D
     / \
    B   C

1.2 Acciones semánticas de un analizador sintáctico

Son fragmentos de código que se ejecutan cada vez que la gramática de un compilador reconoce una regla específica. Sirven para evaluar atributos, generar el árbol sintáctico, o registrar información.

// Ejemplo en YACC/Bison
expr : expr '+' term  { $$ = $1 + $3; } // Acción semántica de suma
     | term           { $$ = $1; };

1.3 Comprobaciones de tipos en expresiones

Verifica que los operandos de una expresión sean compatibles con el operador aplicado (Ej. no se puede sumar un entero y una cadena sin un cast previo). Si los tipos difieren, el compilador debe inyectar una conversión implícita (coerción) o lanzar un error de tipo.

int x = 5;
float y = 3.2;
// La comprobación deduce que x + y es de tipo flotante.
float result = x + y; // int to float coerción

1.4 Pila semántica en un analizador sintáctico

Es una estructura de datos LIFO paralela a la pila del analizador (pila de estados). En ella se guardan los atributos semánticos de los símbolos no terminales a medida que se va construyendo el árbol. Cuando ocurre una reducción matemática, se extraen elementos de la pila semántica para evaluarlos.

// Pila de estados/símbolos: [0, id, 4, +, 7]
// Pila semántica LIFO:  [val_a, val_b]
// Al reducir Regla (E -> E + T), los topes numéricos se operan.

1.5 Esquema de traducción

Un esquema de traducción dirigida por la sintaxis es una gramática libre de contexto en donde se han integrado variables y fragmentos de código. Permite asociar atributos a los símbolos y evaluar propiedades.

// Esquema de traducción inyectado a una gramática para suma
E → E_1 + T  { E.val = E_1.val + T.val }
T → num      { T.val = num.lexval }

1.6 Generación de la tabla de símbolo y tabla de direcciones

La tabla de símbolos almacena las declaraciones de variables, funciones y clases, incluyendo su tipo y alcance (scope).

La tabla de direcciones asigna direcciones o desplazamientos (offsets) en memoria para cada elemento presente en la tabla de símbolos.

/* Tabla de Símbolos Conceptuales */
Nombre  | Tipo  | Ámbito | Offset
--------|-------|--------|-------
x       | int   | main   | 0
y       | float | main   | 4
calc    | func  | global | NULL

1.7 Manejo de errores semánticos

Los errores semánticos suceden cuando el código tiene sentido sintácticamente pero carece de sentido lógico para el lenguaje. Ejemplos incluyen el uso de variables no declaradas o funciones invocadas con parámetros incorrectos. El analizador debe recuperar de estos errores para continuar emitiendo diagnósticos.

// Error: type mismatch
int edad = "Veinte"; // Error semántico

Unidad 2: Generación de Código Intermedio

El código intermedio es un puente entre el analizador semántico y el generador de código ensamblador (código máquina). Nos permite tener cierta independencia de la arquitectura y facilitar rutinas de optimización.

2.1 Notaciones

Existen diversas formas de escribir las sentencias para organizarlas durante el proceso de compilación sin la ambigüedad que pueden tener los lenguajes altos.

2.1.1 Prefija, 2.1.2 Infija, 2.2.3 Postfija

  • Infija: El operador está entre operandos. Ej: A + B. (Común en humanos, problemática para máquinas por precedencias).
  • Prefija: El operador precede a los operandos. Ej: + A B.
  • Postfija: El operador se encuentra después. Ej: A B +. (Muy fácil de procesar utilizando una pila).

2.2 Representaciones de código Intermedio

2.2.1 Notación Polaca

Es un sinónimo de la notación Postfija diseñada por el lógico Jan Łukasiewicz. Elimina por completo la necesidad de usar paréntesis matemáticos para dictar la precedencia.

// Expresión Infija natural: (A + B) * C
// Notación Polaca Inversa:  A B + C *

2.2.2 Código P

Es el lenguaje máquina para una supercomputadora teórica o una máquina de pila (Stack Machine). Fue popularizado por el compilador Pascal a nivel intermedio (P-Code).

; Código P para: x = y + z
LOD 1, y   ; Carga y en el tope de la pila
LOD 1, z   ; Carga z encima de y
ADD        ; Suma el tope y subtope
STO x      ; Almacena el resultado en x

2.2.3 Triplos

Los triplos son estructuras de código intermedio utilizadas para representar expresiones mediante tres campos lógicos: (Operador, Operando1, Operando2). A diferencia de otras estructuras, el triplo no usa variables temporales explícitas para guardar sus resultados. En su lugar, el resultado de la operación se define implícitamente por el número de la posición o línea de la instrucción. Las sentencias futuras simplemente referencian esa misma línea para usar su valor calculado.

// Expresión de ejemplo: x = y * z + y * z
// Formato Triplo: Referencias como (1) apuntan al resultado de esa línea
1: ( *, y, z )
2: ( *, y, z )
3: ( +, (1), (2) )
4: ( =, x, (3) )

Desventajas: Mover o reorganizar las instrucciones durante las fases de optimización es excepcionalmente complicado; alterar el orden alteraría el índice de todas las referencias de línea siguientes.

2.2.4 Cuádruplos

Los cuádruplos, como su nombre lo indica, se componen de cuatro campos principales: (Operador, Operando1, Operando2, Resultado). A diferencia de los triplos, en el caso de los cuádruplos el resultado siempre se almacena de manera explícita en una variable temporal real e independiente (por ejemplo, t1, t2).

// Misma expresión de ejemplo: x = y * z + y * z
// Formato Cuádruplo:
1: ( *, y, z, t1 )
2: ( *, y, z, t2 )
3: ( +, t1, t2, t3 )
4: ( =, t3, null, x )

Ventajas: Aunque generan más variables temporales, este código intermedio es altamente reubicable. Al no depender de la línea donde son declaradas, las ramas de optimización pueden manipular y mover las instrucciones sin destruir el árbol de dependencias.

2.3 Esquema de generación

Son un conjunto de directrices algorítmicas de cómo transformar el código fuente en código intermedio.

2.3.1 Variables, Constantes y 2.3.2 Expresiones

Transforman todas las cargas lógicas a variables temporales, respetando el tipo y las comprobaciones previamente evaluadas por el analizador semántico.

// Transformación de expresión compleja: x = (a * b) + c
t1 = a * b
t2 = t1 + c
x = t2

2.3.3 Instrucción de asignación y 2.3.4 Control

Las sentencias de control (como el IF/ELSE y el WHILE) no existen rígidamente en ensamblador. Se modelan usando saltos condicionales y temporales 1D y etiquetas lógicas (goto L1).

// Código original: if (a < b) then x = 1 else x = 0;
100: if a < b goto 103
101: x = 0
102: goto 104
103: x = 1
104: ... (siguiente código)

2.3.5 Funciones y 2.3.6 Estructuras

Las funciones y rutinas en código intermedio requieren de instrucciones específicas para el paso de parámetros y rescate de la memoria (Stack Frame / Contexto local de la función). Las estructuras se fragmentan linealizando sus componentes sumando desplazamientos precalculados a la dirección base desde el paso de la Unidad 1.6.

// Estructuración de parámetros en llamadas: draw(x, y)
param x
param y
call draw, 2

Unidad 3: Optimización

La optimización de código es la fase que busca mejorar el código intermedio (o final) para que el programa consuma menos recursos (memoria, tiempo de ejecución) manteniendo la misma funcionalidad y resultados.

3.1 Tipos de optimización

3.1.1 Locales

Se aplican dentro de un bloque básico de código (una secuencia de instrucciones sin saltos). Incluyen:

  • Plegamiento de constantes: Evaluación de expresiones constantes en tiempo de compilación. Ej: x = 2 + 3 se cambia a x = 5.
  • Propagación de constantes: Reemplazo de variables por su valor constante conocido.
  • Reducción de fuerza: Reemplazar operaciones costosas por otras más baratas (e.g., multiplicar por 2 se cambia a desplazamiento de bits o suma).
  • Eliminación de subexpresiones comunes: Evitar recalcular expresiones que ya fueron calculadas en el mismo bloque.
// Plegamiento y Propagación de constantes en bloque local
// Original:             // Optimizado:
x = 2 * 3;             x = 6;
y = x + 0;             y = 6;

3.1.2 Ciclos

Dado que los programas pasan la mayor parte de su tiempo en ciclos (bucles), estas optimizaciones son críticas para el rendimiento global:

  • Movimiento de código (Invariantes de bucle): Sacar del ciclo operaciones cuyo resultado no cambia en cada iteración.
  • Desenrollado de bucles (Loop unrolling): Reducir la sobrecarga del control del ciclo, procesando múltiples iteraciones secuencialmente a la vez.
  • Fusión de bucles: Combinar múltiples bucles adyacentes que iteran sobre el mismo rango en un solo bucle.
// Movimiento de código invariante de ciclo
// Antes:
while (i < n) { 
  x = a + b; // 'a + b' No cambia nunca dentro del ciclo
  i++;
}
// Después:
x = a + b;
while (i < n) { i++; }

3.1.3 Globales

Más allá de un único bloque, analizando el grafo de flujo de control del programa o nivel función. Permiten realizar propagación de constantes global, identificar variables vivas, y la eliminación de código muerto o inalcanzable.

// Eliminación de código inalcanzable analizado globalmente
int suma() {
  return 5;
  int x = 10; // Después de un return esto nunca se ejecutará. Se omite.
}

3.1.4 De mirilla (Peephole)

Es una técnica que inspecciona mecánicamente reducidas secuencias de código objeto (la "mirilla") para reemplazarlas por una secuencia más eficiente de la máquina. Ejemplos:

  • Eliminación de instrucciones redundantes de carga o almacenamiento en memoria.
  • Reducción de saltos sobre saltos o hacia la siguiente instrucción.
; Código original
MOV R1, R2
MOV R2, R1  ; Esta instrucción se elimina por redundante

; Optimización de salto a salto
JMP L1
...
L1: JMP L2  ; La mirilla cambia el primero directamente por JMP L2

3.2 Costos

3.2.1 Costo de ejecución (memoria, registros, pilas)

Medir el impacto y recursos en tiempo de compilación o de ejecución. Optimizar implica minimizar el uso según las necesidades del contexto.

  • Registros: Son la memoria más rápida del CPU; el algoritmo de Asignación de Registros minimiza los volcados en la pila de memoria (RAM) acelerando el programa.
  • Memoria: Mejorar la localidad espacial y temporal para aprovechar adecuadamente la memoria caché.
  • Pilas: Evitar realizar llamadas excesivas e innecesarias a rutinas (overheads de stack), posiblemente reemplazando llamadas por código embebido (Inline expansion) en donde sea eficiente.

3.2.2 Criterios para mejorar el código

Se busca centrar la optimización en las partes del código que consumen la mayor cantidad del tiempo de ejecución para tener beneficio comprobable (Regla de Pareto, Hotspots). El código resultado optimizado debe ser semánticamente idéntico al original, no alterar una salida que es válida, justificar el tiempo o espacio que demora realizarse la misma optimización, y no introducir nuevos riesgos de estabilidad.

3.2.3 Herramientas para el análisis del flujo de datos

Para aplicar las transformaciones es necesario extraer propiedades estables mediante Análisis de Flujo de Datos (Data-Flow Analysis). Utiliza Grafos de Flujo para deducir propiedades como:

  • Variables Vivas: Datos calculados que podrían ser leídos en el futuro (si no están vivas, se descartan).
  • Definiciones Alcanzables: Asignaciones de valores que logran llegar sin ser sobreescritas hasta un cierto punto.

La recolección de este conocimiento de los diferentes caminos y ramificaciones permite saber cuándo es matemáticamente seguro aplicar qué optimizaciones sin cambiar el comportamiento del programa.

Unidad 4: Generación de código objeto

La generación de código objeto es la fase final del compilador. Recibe como entrada el código intermedio (optimizado o no) y su objetivo principal es producir de salida un código equivalente en el lenguaje de la máquina objetivo (código objeto o ensamblador), teniendo siempre en cuenta la arquitectura física y los recursos del procesador en cuestión.

4.1 Registros

La correcta asignación y administración de los registros es vital. Como los registros del procesador central son la memoria más veloz y más escasa, el compilador debe decidir cuidadosamente qué variables y resultados temporales deben residir en registros y cuáles deben volcarse de vuelta en la memoria principal.

; Ejemplo de asignación a registros en x86
MOV EAX, 10    ; Se asigna el valor 10 al registro EAX
ADD EAX, EBX   ; Se suma EBX a EAX sin acceder a la RAM externa

Existen algoritmos robustos como el "coloreado de grafos", donde se mapean y deducen las interferencias de variables vivas para aprovechar al máximo los registros finitos evitando sobrescribir información importante.

4.2 Lenguaje ensamblador

El compilador suele generar primero un archivo en lenguaje ensamblador en lugar de escribir código máquina puro. El ensamblador es un set de instrucciones legibles empleando reglas nemotécnicas (como ADD, MOV, JMP) que son altamente dependientes de la familia del procesador a usar (x86, ARM, RISC-V).

; Ejemplo de una suma (a = b + c) en ensamblador ARM
LDR r1, [b]    ; Carga de la RAM el valor 'b' en el registro 1
LDR r2, [c]    ; Carga de la RAM el valor 'c' en el registro 2
ADD r0, r1, r2 ; Suma r1 y r2, y se guarda el resultado en r0
STR r0, [a]    ; Almacena el nuevo registro 0 de vuelta en 'a'

4.3 Lenguaje máquina

El lenguaje máquina es directamente la secuencia de código fuente binaria (ceros y unos) comprensible y ejecutable por la tarjeta madre y procesador. Usualmente es otro subprograma independiente llamado Ensamblador (Assembler) el encargado del último salto de traducción del código lógico de nemotécnicos a los impulsos de instrucciones y números absolutos.

; Representación de instrucciones pasadas a código máquina (mostrado en hex)
B8 0A 00 00 00  ; Codificación de instrucción general: MOV EAX, 10
01 D8           ; Codificación de instrucción general: ADD EAX, EBX

4.4 Administración de memoria

Para construir el programa final, el compilador debe indicar cómo la vida de variables creadas mapearán e instanciarán bloques en RAM. El enlazador aparta zonas estructurales delimitadas que tendrá el programa para cada ejecución:

  • Texto / Código (Code Segment): Espacio de memoria de sólo lectura que aloja las macros y rutinas compiladas.
  • Datos (Data y BSS Segments): Espacios persistentes para guardar variables globales, literales inmutables y estáticos.
  • Pila de Llamada (Stack): Espacio para el flujo de rutinas anidadas (Stack Frames) de las funciones y conteniendo a todas sus variables localmente efímeras. Su base incrementa automáticamente controlando la pila.
  • Montículo (Heap): Área designada sin métricas en un principio donde radicará la persistencia de los punteros lógicos que irán solicitando espacio el tiempo de marcha o compilación en sí (funciones como malloc() o el operador new).
// Mapeo figurado en entorno estilo lenguaje C
int global_var = 10; // Se reserva directamente en BSS / Data
void func() {
  int local_var = 5; // Se apila temporalmente en el Stack
  int* p = malloc(4); // Solicita en tiempo de ejecución en el Heap
}
